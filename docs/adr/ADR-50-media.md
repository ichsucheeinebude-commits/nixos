# ADR-50: Domain Media Architecture

> Complete media stack: library management, automated download (*arr suite via factory), streaming (Jellyfin/Navidrome/Audiobookshelf), discovery (Jellyseerr), and hardware-accelerated transcoding via Intel QuickSync.

---

## Context

Domain 50 implements a full self-hosted media solution on the Q958 (Intel i3-9100 with QuickSync). It covers media library paths, the *arr automation suite (Sonarr, Radarr, Prowlarr, Lidarr), download clients (SABnzbd), streaming servers (Jellyfin, Navidrome, Audiobookshelf), and media request management (Jellyseerr). All media services share GID 169 for unified file access across tiers.

---

## Decisions

### 50-50: Media Library
**Decision:** Centralized media paths with shared GID 169 for all media services. Read-only mounts for libraries. VPN confinement for download stack.
**Rationale:** Shared GID ensures consistent file permissions across all *arr services. Read-only mounts protect library integrity. VPN confinement prevents download leaks.
**Alternatives considered:** Per-service users (rejected — permission conflicts).

### 50-51: Arr Factory (nixflix + nixarr patterns)
**Decision:** Single factory function (`mkArrService`) generates all *arr services (Sonarr, Radarr, Lidarr, Prowlarr, Readarr). Shared patterns: state management in `/data/.state/nixarr/<service>`, systemd hardening (ProtectSystem=strict, ProtectHome, NoNewPrivileges, PrivateTmp), theme.park integration, TRaSH-Guides quality profiles (1080p H.265, good file size), RestartTriggers for theme-park package. Per-service overrides: port, subdomain, rootFolders, quality profile params. Backup exclusion for all Arr state (re-downloadable).
**Rationale:** Eliminates ~800 Lines of redundant code across 6 separate modules. Single source of truth for *arr patterns. Easy to add new *arr services (e.g., Bazarr) by calling `mkArrService` with custom params.
**Alternatives considered:** Separate modules per service (rejected — 80% copy-paste, maintenance nightmare).

### 50-52: Download Stack
**Decision:** SABnzbd for NZB downloads. WireGuard namespace confinement — no leak possible.
**Rationale:** NZB is faster and more reliable than torrents for most content. WireGuard confinement prevents IP leaks.
**Alternatives considered:** qBittorrent-only (rejected — NZB is preferred), no VPN (rejected — IP exposure risk).

### 50-53: Streaming Stack (upgraded)
**Decision:** Unified streaming stack module: Jellyfin (QuickSync HW transcoding, plugin management, auto-libraries from Arr root folders), Navidrome (music streaming, Subsonic API), Audiobookshelf (audiobooks + podcasts). Shared patterns: state management in `/data/.state/nixarr/<service>`, systemd hardening, theme.park integration. Jellyfin cache/transcodes excluded from backup (regenerable).
**Rationale:** Single module for all streaming services reduces complexity. QuickSync offloads transcoding from weak i3 CPU. Navidrome/Audiobookshelf share same hardening patterns.
**Alternatives considered:** Separate modules per service (rejected — shared patterns, simpler to maintain).

### 50-54: Media Discovery
**Decision:** Jellyseerr for self-hosted media request management. Family-friendly request workflow. Integration with Sonarr/Radarr.
**Rationale:** Jellyseerr provides a user-friendly request portal for family members. Automates the request → download workflow.
**Alternatives considered:** Ombi (rejected — Jellyseerr is more actively maintained).

### 50-55: Jellyfin (consolidated into 53-streaming-stack)
**Decision:** Jellyfin moved into unified streaming stack. QuickSync HW transcoding (iHD driver), plugin management (AniDB, AniList, TMDB, TVDB, OpenSubtitles), theme.park integration, auto-library generation from Arr root folders. State in `/data/.state/nixarr/jellyfin/`. Cache and transcodes excluded from backup.
**Rationale:** Consolidated with Navidrome/Audiobookshelf into single module. Shared patterns reduce redundancy.
**Alternatives considered:** Standalone module (rejected — redundant with other streaming services).

### 50-56: Sonarr (consolidated into 50-mkarr-factory)
**Decision:** Sonarr generated by `mkArrService` factory. TRaSH-Guides quality profile (1080p H.265, max 4GB per episode), theme.park integration, state in `/data/.state/nixarr/sonarr/` (excluded from backup), systemd hardening.
**Rationale:** Factory eliminates redundant code. H.265 provides smaller files at good quality.
**Alternatives considered:** Separate module (rejected — 80% copy-paste with other *arr services).

### 50-57: Radarr (consolidated into 50-mkarr-factory)
**Decision:** Radarr generated by `mkArrService` factory. TRaSH-Guides quality profile (1080p H.265, 500MB-8GB), theme.park integration, state in `/data/.state/nixarr/radarr/` (excluded from backup).
**Rationale:** Factory eliminates redundant code. H.265 provides 3-6GB typical for 1080p movies.
**Alternatives considered:** Separate module (rejected — 80% copy-paste).

### 50-58: Prowlarr (consolidated into 50-mkarr-factory)
**Decision:** Prowlarr generated by `mkArrService` factory. Declarative settings sync, SceneNZB.com as sole indexer (Newznab API v1, REST), theme.park integration, state in `/data/.state/nixarr/prowlarr/` (excluded from backup).
**Rationale:** Single authoritative indexer reduces complexity. SceneNZB provides high-quality NZB content.
**Alternatives considered:** Separate module (rejected — 80% copy-paste).

### 50-59: Lidarr (consolidated into 50-mkarr-factory)
**Decision:** Lidarr generated by `mkArrService` factory. TRaSH-Guides quality profile (lossless preferred), theme.park integration, state in `/data/.state/nixarr/lidarr/` (excluded from backup).
**Rationale:** Lossless music provides best quality. Factory eliminates redundant code.
**Alternatives considered:** Separate module (rejected — 80% copy-paste).

## Consequences

### Positive
- Complete automated media pipeline: request → download → organize → stream
- Hardware transcoding enables 4K playback on weak CPU
- Shared GID 169 eliminates permission issues across services
- VPN-confined downloads prevent IP exposure
- Single toggles reduce configuration complexity

### Negative
- Media stack is the most resource-intensive domain
- Indexer availability depends on external sources
- Download stack VPN adds latency
- Large media library requires careful storage management (Domain 30)
